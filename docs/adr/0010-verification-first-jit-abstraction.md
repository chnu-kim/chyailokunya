# ADR-0010: 원칙 — 검증 가능성 우선 · JIT 추상화(YAGNI)

- 상태: Accepted
- 날짜: 2026-07-18

## 맥락

이 프로젝트를 관통하는 메타 결정이다. 개별 기술 선택([ADR-0001](./0001-next-app-router.md)~
[ADR-0009](./0009-actions-gate-workers-builds.md))이 왜 그렇게 정렬됐는지의 근거.

## 결정

두 가지를 협상 불가 원칙으로 둔다.

1. **검증 가능성 우선.** TS strict · Vitest · Playwright · dependency-cruiser 경계 ·
   Drizzle 마이그레이션 검증이 **CI 게이트**로 나쁜 머지를 막는다. 눈으로 판단하지 않고
   기계가 강제한다.
2. **JIT 추상화(YAGNI).** 추상화 레이어는 **실제 두 번째 기능이 seam 을 드러낼 때** 넣는다.
   미리 만든 일반화는 대개 틀린 일반화다.

## 근거

- 빌드·타입·테스트가 없던 정적 사이트에서 왔다 — 그때 "검증을 대신해 줄 게 없다"가
  반복해서 물렸다. 새 스택의 값어치는 그 검증을 코드로 붙박는 데 있다.
- 팬사이트는 요구가 유동적이다. 이른 추상화는 잘못된 방향으로 굳는다. 경계
  ([ADR-0007](./0007-single-app-enforced-boundaries.md))는 강제하되, 레이어 안의 구조는
  필요가 증명될 때 자란다.

## 기각한 대안

- **선행 설계 완비 후 구현** — 유동적 요구에서 과설계로 굳는다.
- **검증 나중에** — 게이트 없는 속도는 곧 회귀로 되돌아온다. 이 프로젝트에서 이미 겪었다.

## 결과

- (+) 모든 결정이 "이게 검증 가능한가 / 지금 필요한가"라는 한 잣대로 평가된다.
- (−) 어떤 편의(캐시 백엔드·범용 API 레이어·공유 패키지)는 일부러 비워 둔다 — 자리표시자와
  빈 레이어가 보이는 건 미완이 아니라 의도다.
