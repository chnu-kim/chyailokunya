# ADR-0004: API 는 tRPC + Zod

- 상태: Accepted
- 날짜: 2026-07-18

## 맥락

클라이언트↔서버 데이터 계약이 필요하다. 원칙은 "계약(스키마·라우터)이 타입 정본",
그리고 "AI 중심 유지보수 — 경계는 기계 강제". 단일 앱·단일 배포([ADR-0007](./0007-single-app-enforced-boundaries.md))
안에서 프론트와 백엔드가 같은 타입을 공유한다.

## 결정

**tRPC** 로 프로시저를 정의하고 입력은 **Zod** 로 검증한다. 라우터 타입이 클라이언트로
그대로 흘러 들어가 별도 API 스키마 동기화가 필요 없다.

## 근거

- 단일 TS 레포라 tRPC 의 "빌드 없는 end-to-end 타입"이 그대로 산다 — REST/OpenAPI
  코드젠 왕복이 사라진다.
- Zod 입력 스키마가 신뢰 경계(클라이언트·localStorage·OAuth 콜백)에서 런타임 검증을
  강제한다. 구 사이트에서 localStorage 를 신뢰하지 않는 입력으로 다룬 규율의 연장.
- Drizzle 스키마 타입([ADR-0003](./0003-d1-drizzle.md))과 Zod 를 한 계약으로 잇기 쉽다.

## 기각한 대안

- **REST + 수작업 타입** — 계약이 두 곳(서버·클라)에 갈려 표류한다.
- **GraphQL** — 스키마/리졸버/코드젠 오버헤드가 팬사이트 규모에 과하다.
- **서버 액션만** — 뮤테이션엔 편하나, 공개 읽기·입력 검증·라우터 조직화에서 tRPC 의
  구조가 낫다. 필요하면 병행 가능.

## 결과

- (+) 프론트/백 타입이 컴파일 타임에 한 몸으로 묶인다.
- (+) 모든 입력이 Zod 경계를 통과 — 인가 가드([ADR-0012](./0012-role-based-writes-allowlist.md))의
  앞단이 된다.
- (−) tRPC 는 Next/서버 런타임에 결합된다 — 외부 공개 API 가 필요해지면 그때 별도 레이어를
  JIT 로 얹는다.
