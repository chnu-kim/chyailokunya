# ADR-0020: 게임 삭제 = 확인 다이얼로그 (지연 커밋 되돌리기 창 폐기)

- 상태: Accepted
- 날짜: 2026-07-20
- 뒤집음: [ADR-0014](./0014-v1-data-model-schema.md)의 "되돌리기는 클라이언트 지연 커밋" — 삭제 확인은 이제 행동 **전**에 온다.
- 유지: 같은 절의 **하드 삭제**(`deleted_at` 없음) 결론은 그대로다. 근거만 갈아 끼운다.

## 맥락

[ADR-0014](./0014-v1-data-model-schema.md)는 게임 삭제를 **지연 커밋**으로 설계했다. 삭제
버튼을 누르면 카드가 그 자리에서 ghost 로 바뀌고, 타이머(구현값 6초)가 만료되기 전까진 `remove`
뮤테이션을 아예 보내지 않는다. 그 창 안에 "되돌리기"를 누르면 없던 일이 된다.

그 설계가 실제로 요구한 것은 카드 한 장짜리 UI 가 아니라 **상태 기계**였다:

- 카드마다 ghost 상태(`GhostState`)와 타이머 핸들을 들고, 언마운트·중복 삭제·연속 삭제를 다 정리해야 한다.
- 삭제된 카드가 DOM 에서 사라지므로 **포커스를 누구에게 넘길지**를 손으로 정해야 한다(되돌리기
  버튼 → 만료 후엔 추가 슬롯). 포커스 인계가 빠지면 키보드 사용자는 문서 최상단으로 튕긴다.
- 목록의 "표시용 실제 값"이 서버 데이터가 아니라 `games - ghosts` 라는 파생값이 된다. 총 개수·
  빈 상태 분기가 전부 이 파생값에 얹힌다.
- 실패 문구가 사용자 눈에 직접 닿지 않고 ghost 안 announcement 로만 흘러, 삭제 실패가 조용해진다.

그 대가로 얻는 기능은 정확히 하나 — **"6초 안에 마음 바꾸기"** 다.

## 결정

지연 커밋·ghost·되돌리기를 전부 걷어내고 **삭제 확인 다이얼로그**로 바꾼다. 확인하면 즉시
`remove` 뮤테이션이 나간다. 취소하면 아무 요청도 없었던 것과 같다.

다이얼로그는 신규 모달이 아니라 이미 있는 `GameDialog` 를 재사용한다 — 포커스 트랩·복귀가
그 키트에 이미 들어 있어, 폐기하는 포커스 인계 로직을 다시 짜지 않는다.

**하드 삭제는 그대로다.** 스키마 변경이 없고(`deleted_at` 없음), 서버 계약도 그대로다(없는 id →
`deleted: false`).

## 근거

- **같은 방어를 더 싼 값에.** 되돌리기 창과 확인 다이얼로그는 둘 다 "실수로 지웠다"를 막는
  장치다. 다른 건 **언제 막느냐**뿐이다 — 되돌리기는 파괴가 일어난 뒤 6초를 벌고, 확인은
  파괴 전에 멈춘다. 뒤에서 막으려면 되돌릴 상태를 어딘가에 들고 있어야 하고, 앞에서 막으면
  들 상태가 없다. 실수 방어라는 목적이 같다면 상태를 안 남기는 쪽이 이긴다.
- **하드 삭제는 근거를 갈아 끼울 필요가 없다 — 0014 가 댄 이유가 애초에 진짜 이유가 아니었다.**
  0014 는 `deleted_at` 을 안 둔 이유로 "되돌리기가 서버를 안 건드리니 삭제 상태를 영속시킬
  필요가 없다"를 들었지만, 그 문장이 없어도 결론은 선다. 하드 삭제를 실제로 지지하는 건 0014
  자신이 소프트 삭제를 기각하며 댄 두 가지다: **복구 휴지통·삭제 감사 요구가 v1 에 없다**,
  그리고 **모든 조회가 `WHERE deleted_at IS NULL` 을 영구히 지불한다.** 둘 다 이 변경과 무관하게
  그대로 유효하므로 결론도 그대로다.
  한때 이 자리에 "확인이 파괴 전에 오므로 서버에 도달한 삭제는 이미 의도된 삭제다"라고 썼다가
  걷어냈다. **성립하지 않는다** — 반복되는 확인은 습관화되어 통과 클릭이 되고, 그러면 "다이얼로그가
  읽혔다"는 전제가 무너진다. 결론이 옳다고 해서 아무 근거나 매달면 문서가 가장 약한 고리의
  강도를 갖는다.
- **검증 가능성**([ADR-0010](./0010-verification-first-jit-abstraction.md)). 사라지는 것은 타이머·
  ghost 파생 목록·포커스 인계처럼 **테스트하기 가장 비싼 종류의 상태**다. 남는 것은 "확인을
  눌렀나"라는 불리언 하나다.
- **실패가 드러난다.** 즉시 커밋이라 삭제 실패가 처음으로 사용자 눈앞에 뜬다(이전엔 만료 시점에
  조용히 실패할 수 있었다). 문구는 `deleteErrorMessage` 가 맡는다 — 추가·수정용 `writeErrorMessage`
  를 그대로 쓰면 "저장에 실패했어요"가 되어 뜻이 뒤집힌다.
- 사용자 결정이다(2026-07-20). 재논쟁 대상이 아니다.

## 기각한 대안

- **되돌리기 창 유지 + 타이머만 늘리기** — 상태 기계가 문제였지 6초라는 숫자가 문제가 아니었다.
  늘리면 ghost 가 화면에 더 오래 남아 목록 파생값 문제만 커진다.
- **확인 + 되돌리기 둘 다** — 같은 실수를 두 번 막으면서 비용은 양쪽을 다 낸다. 확인을 통과한
  삭제를 또 되돌리게 하는 건 확인 다이얼로그를 무의미한 클릭으로 만든다.
- **소프트 삭제(`deleted_at`)로 전환해 서버 휴지통** — 0014 가 기각한 그대로다. 확인 다이얼로그를
  붙이는 변경이 복구 UI·보존 정책·조회 필터라는 새 요구를 끌고 올 이유가 없다.
- **append-only 아카이브 행** — 삭제할 때 스냅샷을 별도 테이블에 남기고 `games` 는 그대로 하드
  삭제한다. 위 소프트 삭제와 달리 **조회 비용이 0 이다** — `games` 의 읽기 경로에 필터가 한 줄도
  안 붙으므로 0014 의 기각 사유 둘 중 "영구 조회 비용" 항이 통째로 사라진다. 패턴도 가설이
  아니다: [ADR-0018](./0018-role-audit-and-elevation-guard.md) 의 `role_audit_logs` 가 이미 그
  형태고, 삭제도 admin 전용의 드문 행동이라 결이 맞는다.
  **그럼에도 지금은 안 한다** — 남는 기각 사유("복구·감사 요구가 v1 에 없다")가 그대로이기 때문이다.
  아카이브 행은 쓰는 순간부터 보존 기간·누가 볼 수 있나·복구 경로라는 질문을 데려오는데, 그
  질문들을 물을 사람이 아직 없다. **삭제 감사 요구가 생기면 이 대안이 첫 후보다** — 소프트 삭제로
  가지 말고 여기로 온다.
- **`window.confirm`** — 스타일링이 불가해 사이트 크롬과 어긋나고, 접근 이름·포커스 복귀를
  브라우저에 맡기게 된다. `GameDialog` 가 이미 있으므로 새 의존도 아니다.

## 결과

- (+) `GhostState`·`UNDO_MS`·ghost 목록 파생·포커스 인계가 통째로 사라진다.
- (+) 삭제 실패가 사용자에게 보인다.
- (+) 파괴 전에 멈추므로 "되돌리기를 못 본 채 6초가 지나갔다"는 실패 모드가 없어진다.
- (−) 실수 삭제의 복구 수단이 **0** 이 된다. 확인을 통과한 삭제는 끝이다 — 이건 하드 삭제를
  유지하는 한 받아들이는 대가고, 되돌릴 수 없다는 사실이 다이얼로그 문구에 드러나야 한다.
- (−) **같은 변경 묶음이 이 대가를 키운다.** 수정·삭제를 카드 썸네일 우상단 오버레이(44px 버튼 둘,
  hover 로만 노출)로 옮기면서 옆 카드의 버튼을 잘못 누를 확률이 라벨 줄이던 시절보다 올라갔다.
  즉 오식 가능성을 올리면서 오식의 백스톱을 없앤 셈이다. 확인 다이얼로그가 그 백스톱 자리를
  혼자 진다 — 그래서 다이얼로그가 **어느 게임인지**를 반드시 보여야 하고(`describedBy` 로
  이름을 접근성 트리에도 싣는다), 파괴 버튼이 취소와 시각적으로 구분돼야 한다.
- (−) 삭제가 클릭 2회로 늘어난다. 관리자만 하는 드문 행동이라 수용한다.
