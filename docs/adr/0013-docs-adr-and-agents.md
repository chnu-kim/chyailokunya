# ADR-0013: 문서는 ADR(왜) + AGENTS.md(플레이북)

- 상태: Accepted
- 날짜: 2026-07-18

## 맥락

구 사이트는 `AGENTS.md` 를 정본으로, `CLAUDE.md`·`GEMINI.md` 가 그걸 import 하는 구조로
에이전트 지침을 한 곳에 모았다. 그 규율은 효과가 있었다. 새 스택은 결정이 많아졌으므로
"규칙"과 "왜"를 분리해 관리한다.

## 결정

- **`docs/adr/*` = 결정의 "왜".** 굵직한 아키텍처 결정마다 ADR 하나. append-only, 뒤집을
  땐 Superseded 로 표시하고 새로 쓴다.
- **`AGENTS.md` = 플레이북.** 아키텍처 맵 + feature 추가 절차 + 불변식(협상 불가 규칙).
  `CLAUDE.md`·`GEMINI.md` 는 이를 import 한다.
- **계약이 타입 정본.** Drizzle 스키마·tRPC 라우터·Zod 가 "무엇"의 정본이고, 문서는 그걸
  중복 서술하지 않는다.

## 근거

- 규칙(AGENTS.md)과 근거(ADR)를 나누면 각자 수명이 다르게 관리된다 — 규칙은 갱신되고,
  결정 기록은 역사로 쌓인다.
- 코드가 "무엇"을 이미 말하므로(타입 계약), 문서는 코드가 말 못 하는 "왜"와 "하지 말 것"에
  집중한다. 구 사이트의 "주석은 제약을 적는다" 규율의 연장.

## 기각한 대안

- **README 한 곳에 다 적기** — 결정·규칙·사용법이 뒤엉켜 아무도 최신으로 못 지킨다.
- **문서 없이 코드로만** — "왜 D1 이고 왜 Pages 를 버렸나" 같은 건 코드에 안 남는다.

## 결과

- (+) 새 에이전트/기여자가 규칙은 AGENTS.md, 근거는 ADR 로 빠르게 잡는다.
- (+) 결정을 뒤집을 때 역사가 보존된다.
- (−) ADR 을 쓰는 규율이 필요하다 — 굵직한 결정에서 ADR 을 빼먹으면 "왜"가 다시 증발한다.
